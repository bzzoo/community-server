### 도메인 응집에 대한 방향성
- DDD 설계에 대한 회의론
  - 애그리거트 루트의 비대성
  - 모델의 높은 자유성
  - ORM 엔티티와 어플리케이션 모델의 비정합성
  - DDD 또는 EDA 설계의 오버 엔지니어링

- 헥사고날 아키텍처에 대한 회의론
  - 과도한 계층간 매핑
  - 과도한 인터페이스, 어댑터의 분리
  - 도메인간 소통비용 증가
  

### 어떻게 응집할 것인가?
- 모노리스 어플리케이션에서도 추가 기술 없이 충분한 도메인 응집이 가능하다.
 - 계층간 규칙과 의존관계를 명확히 한다.
 - 모듈간 의존관계를 명확히 한다.
 - 최상위 추상계층과 그 하위 구현계층을 추가한다.


### 뭐가 좋은가?
- 도메인 응집을 표상하는 아키텍처들에 뒤치지 않게 도메인 응집을 할 수 있다.
- 도메인 모델이 비대해지지 않으며 간략하고 수정과 확장이 빠르다.
- 응집 수준이 중간 정도의 강도를 갖고 있기 때문에 레이어드[수평설계] 에서 기능 기반 컴포넌트 아키텍처[수직설계]로 변환이 쉽다.
